## Open questions

0. **Spec 04b — pre-resolved design choices** (recorded 2026-05-04 during the Spec 04 session, before the spec body is drafted): (a) `Note` becomes fully immutable — all fields `final`, hand-written `copyWith` / `==` / `hashCode`; **no** `freezed` (Spec 05 explicitly defers freezed adoption). (b) Migration trigger: only when the legacy `notes_v2` box has at least one entry AND the new `notes_v3` box is empty — idempotent, near-zero cost on warm boots; the legacy box is deleted from disk after a successful migration. So an entry like `lib/services/llm_download_service.dart` would silently exempt every file named `llm_download_service.dart` regardless of directory. Latent today (allowlist is empty); must be fixed before Spec 19 (llm-model-download) adds the first entry. Suggested fix: post-filter the `HITS` text against absolute allowlist paths, or replace `grep --exclude` with `find … -not -path …` followed by `xargs grep`.
11. **`LocalNotificationService` instance-ification** (Spec 05 follow-up): the service is currently a class with `static` methods. `NotesListBloc` injects `cancelNotification` as a constructor callback (defaults to `LocalNotificationService.cancelNotification`) to keep tests honest. A future spec should convert the service to an instance class injected via `RepositoryProvider`, at which point the callback seam in `NotesListBloc` collapses into a regular dependency. Track alongside the notification-id refactor (item 12).
12. **Index-based notification cancellation** (Spec 05 follow-up): `LocalNotificationService.cancelNotification(int id)` takes the **note's current list index** as the notification id, inherited from the legacy `notes_provider.deleteNote`. This is fragile — index volatility post-deletion can mis-cancel reminders. The notifications spec should refactor to id-based or stable-key-based cancellation; until then, `NotesListBloc._onNoteDeleted` cancels before mutating state to preserve current behavior.
